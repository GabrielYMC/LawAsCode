# LawAsCode — 淡江大學學生會數位法制平台

## 壹、問題分析

### 現況痛點

| 問題 | 現狀 | 後果 |
|------|------|------|
| 版本混亂 | 法規以 Word 檔散落各處，無統一版本來源 | 議員拿舊版條文開會、秘書處反覆確認版本 |
| 傳承斷層 | 修法理由隨幹部離職消失 | 後屆無從理解前屆為何如此修訂，重複犯錯 |
| 權力靠人不靠系統 | 「誰能公布法規」由人為判定，非技術權限 | 程序瑕疵難以事後稽核 |
| 參與門檻高 | 查閱法規需翻百頁 PDF，用語生澀 | 一般學生對學生自治望而生畏 |
| 秘書處負擔重 | 收文、排案、發文、排版全靠人力 | 學習門檻高，人因失誤頻繁 |

### 核心洞察

這些問題的共同根源：**法規以「文件」而非「資料」存在**。文件需要人搬運、人判讀、人排版；資料可以被程式處理、被權限保護、被版本追蹤。

## 貳、解決方案

以四個轉型方向回應痛點：

| 轉型方向 | 說明 |
|----------|------|
| **法規資料化** | 法規轉為結構化的 Akoma Ntoso (AKN) XML，每條文有唯一識別碼 |
| **行政自動化** | 修法流程映射至 Git 工作流，收文、排案、發文由系統驅動 |
| **權力權限化** | 「公布施行」不再是口頭授權，而是系統帳號權限 |
| **介面 AI 化** | 一般學生透過自然語言即可查詢法規、理解權益 |

### 設計原則：不可繞過與完整可追溯

本系統的核心精神：**所有法規變更都追查得到權力的授予，並且無論什麼變更都會留下紀錄。**

這不是功能需求，而是系統的不可違反約束（invariant）。所有架構與實作決策都必須服從以下原則：

| 原則 | 說明 | 實作方式 |
|------|------|----------|
| **唯一變更路徑** | 法規的任何變更只能透過 PR merge 進入 `main` 分支，不接受任何使用者直接對法規資料 CRUD | Gitea branch protection：禁止直接 push to main、禁止 force push、禁止刪除分支 |
| **權力必須授予** | 每一次狀態轉移（排案、交付審查、表決、公布）都必須由具備對應角色的帳號執行 | BFF server routes 驗證角色權限，Gitea API token 綁定角色，前端不直接接觸 Gitea |
| **變更必留紀錄** | 誰在什麼時候做了什麼操作、基於什麼依據（投票結果/議長裁定），全部不可刪除地記錄 | Git commit history + PR timeline + PocketBase audit log，三層紀錄互相印證 |
| **歷程不可竄改** | 已經發生的紀錄不允許被修改或刪除，即使是系統管理員 | 禁止 force push（保護 Git 歷史）、PocketBase 審計表設為 append-only |
| **系統即程序** | 議事程序不再依賴人的自律，而是由系統的狀態機強制執行 | `workflow.ts` 定義合法轉移，server 端拒絕不合法的狀態變更請求 |

**具體的 Gitea 分支保護設定（部署時執行）：**

```
# Gitea Branch Protection API
POST /api/v1/repos/{owner}/{repo}/branch_protections
{
  "branch_name": "main",
  "enable_push": false,              # 禁止直接 push
  "enable_push_whitelist": false,     # 無白名單例外
  "require_signed_commits": false,    # 視需求啟用
  "enable_merge_whitelist": true,     # 只有指定角色可 merge
  "merge_whitelist_usernames": ["president", "secretariat-auto"],
  "enable_approvals_whitelist": true,
  "approvals_whitelist_usernames": ["speaker"],
  "required_approvals": 1,           # 至少需要議長 approve
  "block_on_rejected_reviews": true,  # 有反對意見時阻擋
  "block_on_outdated_branch": true,   # 分支過期時阻擋
  "dismiss_stale_approvals": true     # 程式碼變更後舊 approval 失效
}
```

**Web UI 層的守則：**

- SvelteKit server routes（BFF）是唯一與 Gitea API 互動的入口
- 前端使用者永遠不直接接觸 Gitea
- 每個 API call 在 server 端驗證「當前使用者角色 × 提案當前狀態 → 是否為合法操作」
- 不合法的操作在 server 端直接拒絕（403），而非僅在前端隱藏按鈕

### 技術選型理由

| 元件 | 選擇 | 理由 |
|------|------|------|
| Git 伺服器 | **Gitea** (自架) | 輕量、提供完整 PR/Review API、Go 語言效能佳、MIT 授權 |
| 使用者管理 | **PocketBase** (既有) | 已部署、支援 RBAC、即時訂閱、夠用 |
| 前端框架 | **SvelteKit** | 開發快、SSR 效能佳、適合單人維護 |
| LLM | **Ollama + Gemma 3 12B** (既有) | 已部署、本地運行、零 API 成本 |
| 法規格式 | **AKN XML** | 國際立法標準、結構化、eId 唯一識別、利於 LLM 處理 |
| 中介格式 | **Markdown + YAML frontmatter** | 人類可讀、便於編輯、轉換入口 |

### 為何選 Gitea 而非裸 Git

直接操作裸 Git 倉庫需要自行實作 PR、Review、Label、Webhook 等所有功能。Gitea 開箱即用地提供這些，且有完整 REST API，讓我們專注在業務邏輯（議事流程）而非重造輪子。使用者永遠不會看到 Gitea 介面——他們只透過 LawAsCode 的 Web App 操作。

## 肆、議事流程映射

將《學生議會職權行使辦法》中的實體議事程序，精準映射至 Git 工作流：

### 流程對照表

| 議事階段 | Git 操作 | 系統行為 | 權限控管 |
|----------|----------|----------|----------|
| **提案** | 建立 Branch + 開 PR | 自動檢核連署門檻（章程§48：修章需1/5議員） | 議員、行政部門 |
| **一讀** | PR 標記 `一讀` | 議長宣讀標題（職權行使§6） | **議長** |
| **交付審查** | PR 標記 `委員會審查` | 議長決議交付委員會或逕付二讀（職權行使§6-7） | **議長** |
| **委員會審查** | Review Comments + Suggestions | 逐條討論紀錄自動保存 | 委員會成員 |
| **二讀** | PR 標記 `二讀`，接受新 Commit | 大會逐條討論、實質修正 | 議員 |
| **三讀** | PR 標記 `三讀`，CI 限制修改範圍 | 僅允許文字修正，不得變更實質內容 | 議員 |
| **表決** | 系統內投票（PocketBase 記錄） | 自動計算出席數、同意數、是否達法定門檻 | 出席議員 |
| **公布施行** | Merge to `main` | 自動產生公布命令 PDF（含關防用印） | **僅會長** |
| **逾期未公布** | 系統排程觸發 | 14日未公布，秘書處帳號自動執行 Merge（法規標準規則§5） | 系統自動 |

### 狀態機

```
 提案 ──▶ 一讀 ──▶ 委員會審查 ──▶ 二讀 ──▶ 三讀 ──▶ 待公布 ──▶ 已公布
  │        │         │            │         │         │
  │      (議長)     (委員會)     (大會)    (大會)    (會長)
  ▼        ▼         ▼            ▼         ▼         │
 退回    逕付二讀    退回          退回      退回     14日未公布
                                                    → 秘書處
                                                      自動公布
```

- **議長**控制一讀→委員會/逕付二讀的流向（職權行使§6）
- **委員會**審查後提出審查意見書（議會組織規則§37）
- **大會**表決推進二讀→三讀→待公布
- **會長**執行公布施行（Merge）
- 每次狀態轉移都由系統自動記錄：誰、何時、基於什麼（投票結果/議長裁定），形成完整的數位議事錄

### 三讀保護機制

三讀階段的 CI 檢查邏輯：
- 比對二讀通過版本與三讀 Commit 的 diff
- 僅允許：標點符號修正、錯別字修正、格式調整
- 禁止：新增/刪除條文、修改條文實質內容、變更條號
- 違反時自動阻擋 Commit 並通知提交者

## 伍、Akoma Ntoso 法規資料化

### AKN 結構範例

以《組織章程》第九條為例：

```xml
<act xmlns="http://docs.oasis-open.org/legaldocml/ns/akn/3.0">
  <meta>
    <identification source="#tku-sa">
      <FRBRWork>
        <FRBRthis value="/akn/tw-tku/act/組織章程/main"/>
        <FRBRdate date="2024-06-23" name="公布"/>
        <FRBRauthor href="#會長"/>
      </FRBRWork>
    </identification>
    <lifecycle source="#tku-sa">
      <eventRef eId="e1" date="2015-03-27" type="generation" source="#第34屆議會"/>
      <eventRef eId="e2" date="2024-06-23" type="amendment" source="#第43屆議會"/>
    </lifecycle>
  </meta>
  <body>
    <chapter eId="chp_3">
      <heading>行政</heading>
      <article eId="art_9">
        <heading>正副會長之產生與任期</heading>
        <paragraph eId="art_9__para_1">
          <content>
            <p>本會置會長一名，副會長一名，任期一年，由每年七月一日，
            至翌年六月三十日止，不得連任，其選舉罷免事宜另以法律定之。</p>
          </content>
        </paragraph>
        <paragraph eId="art_9__para_2">
          <content>
            <p>依法出任代理會長、代理副會長，則不受前項連任次數之限制。</p>
          </content>
        </paragraph>
      </article>
    </chapter>
  </body>
</act>
```

### AKN 的價值

| 特性 | 效益 |
|------|------|
| **eId 唯一識別** | `art_9__para_2` 精確引用到「第九條第二項」，跨系統不失真 |
| **metadata 嵌入** | 修訂歷程（lifecycle）直接寫在法規裡，不怕遺失 |
| **結構化查詢** | 程式可直接查「所有提到會長的條文」，不需全文搜尋 |
| **LLM 友善** | 結構化 XML 比 Word 更容易被 AI 正確解析和引用 |
| **國際通用** | 與全球立法資訊系統相容，展示專業度 |

### 轉換策略

```
Word (.docx)  ──parser──▶  Markdown + YAML  ──converter──▶  AKN XML
  (現有法規)                  (中介格式/人工校對)              (最終格式)
```

#### 優先採用既有開源工具

| 階段 | 候選工具 | 說明 |
|------|----------|------|
| AKN 編輯/管理 | **Indigo** (Laws.Africa) | 成熟的開源立法管理平台，原生支援 AKN，可作為 AKN 編輯器或參考實作 |
| AKN 編輯器 | **LIME** (University of Bologna) | 學術界開發的 AKN XML 編輯器，OASIS 官方推薦 |
| AKN Schema 驗證 | **OASIS Akoma Ntoso 3.0 Schema** | 官方 XSD，用於驗證產出的 AKN XML 是否合規 |
| Word 解析 | **mammoth** / **pandoc** | 將 .docx 轉為結構化中介格式 |
| XML 處理 | **lxml** (Python) | 建構與操作 AKN XML |

#### 自建部分（開源貢獻）

1. **中文法規結構解析器**：針對中華民國法規用語（第X條、第X項、第X款、第X目）的自動結構辨識
2. **Markdown→AKN 轉換器**：將校對後的 Markdown + YAML frontmatter 轉為符合 OASIS 3.0 Schema 的 AKN XML
3. 兩個工具皆以 MIT License 開源，作為本專案對中文法制數位化社群的貢獻

### 核心頁面

1. **法規瀏覽器**（公開）
   - 以樹狀結構展示全部法規（章→條→項→款→目）
   - 點選任一條文可查看：現行文字、歷次修訂 diff、修法理由
   - 法規間授權關係視覺化（法規地圖）

2. **議事工作台**（議員登入）
   - 待辦事項：待連署法案、待審查法案、待表決法案
   - 提案介面：選擇要修改的法規 → 在瀏覽器內直接編輯 → 系統自動產生 branch + PR
   - 修正條文對照表：每個提案自動產生，可匯出傳統三欄 PDF
   - 表決介面：顯示法案內容、出席人數、一鍵投票

3. **議長控制台**（僅議長可見）
   - 會議管理：召集大會/臨時大會、設定議程
   - 議案流程控制：一讀宣付、交付委員會/逕付二讀
   - 委員會管理：成員名單、審查進度
   - 主席裁決：可否同數時的裁決介面

4. **會長公布台**（僅會長可見）
   - 待公布法案清單（含逾期倒數計時警示）
   - 「公布施行」按鈕 → 系統 Merge + 自動產生 PDF 命令
   - 公布歷史紀錄

5. **AI 法規顧問**
   - **一般諮詢**（公開）：「社團補助怎麼申請？」→ 引用《預算辦法》§34 回答
   - **修法輔助**（議員登入）：草案衝突檢測、上位法合規檢查、修法歷史脈絡查詢
   - **法規健檢**（提案時自動觸發）：掃描母子法衝突、用語不一致、與中華民國行政法規之牴觸

### 修正條文對照表

修正條文對照表是學校與議會最常用的審議文件格式。系統從 Git diff + AKN 結構自動產生。

#### 傳統格式（PDF 輸出，供列印/公告欄）

以 AKN eId 對齊新舊版本，自動產生三欄對照表：

| 修正條文 | 現行條文 | 說明 |
|----------|----------|------|
| 九、凡社團**成員**，有下列情形之一者... | 九、凡社團**負責人**，有下列情形之一者... | 一、文字修正。二、將負責人修訂為成員。 |

**技術實作**：
- 從 Git diff 取得變更的 AKN 條文（以 eId 精確定位）
- 比對新舊版 XML，提取修正/現行文字
- 變更處以**粗體**或<u>底線</u>標示（模仿傳統藍字標示）
- 「說明」欄位預設由提案人填寫（存於 PR description），AI 可輔助建議
- 使用 **docx-js** 產生 .docx → 轉 PDF，或直接用 **Puppeteer** 從 HTML 模板產生 PDF

## 捌、AI 法規顧問架構

AI 顧問不只是對外的查詢工具，也是對內的修法輔助工具。依使用情境分為三種模式：

### 三種 AI 模式

| 模式 | 使用者 | 功能 | 檢索範圍 |
|------|--------|------|----------|
| **一般諮詢** | 一般學生 | 「社團補助怎麼申請？」→ 引用條號回答 | 本會法規 |
| **修法輔助** | 法規委員會、議員 | 草案衝突檢測、用語一致性檢查、立法理由建議 | 本會法規 + 修法歷程 + 中華民國行政法規 |
| **法規健檢** | 提案人 | 上傳修正草案，自動掃描問題 | 本會法規 + 上位法（大學法等） |

### 知識庫架構

```
┌────────────────────────────────────────────────┐
│                   知識庫                        │
│                                                │
│  ┌──────────────┐  ┌───────────────────────┐   │
│  │ 本會法規      │  │ 中華民國相關法規       │   │
│  │ (AKN, eId)   │  │ (大學法、人民團體法、  │   │
│  │ 18部法規      │  │  行政程序法等)         │   │
│  └──────────────┘  └───────────────────────┘   │
│                                                │
│  ┌──────────────┐  ┌───────────────────────┐   │
│  │ 修法歷程      │  │ 釋字/判例             │   │
│  │ (Git log +   │  │ (評議會釋字、         │   │
│  │  PR 討論)    │  │  修法理由)            │   │
│  └──────────────┘  └───────────────────────┘   │
└────────────────────────────────────────────────┘
                    │
                    ▼
             ┌─────────────┐
             │ 向量檢索     │  以條文為單位切片，保留 eId
             │ (Embedding)  │  區分來源（本會/國家法規）
             └──────┬──────┘
                    │
                    ▼
             ┌─────────────┐
             │ Gemma 3 12B │  必須引用具體條號
             │ (Ollama)    │  區分「本會法規」與「參照法規」
             └──────┬──────┘
                    │
                    ▼
              附帶引用來源的回答
```

### 修法輔助模式的具體功能

1. **母子法衝突偵測**：修正《預算辦法》時，自動檢查是否牴觸《組織章程》第七章
2. **上位法合規檢查**：檢查本會法規是否牴觸《大學法》、《人民團體法》等中華民國法規
3. **用語一致性**：掃描全文「會費/經費/費用」等同義詞是否定義統一
4. **歷史脈絡查詢**：「第九條為什麼這樣寫？」→ 從 Git 歷史中找到修訂 PR 及討論
5. **立法理由建議**：根據修正內容，AI 建議修法理由的撰寫方向

### AKN 對 RAG 的優勢

每個條文自帶 eId，AI 回答時可精確引用到項款目層級，使用者可一鍵跳轉到法規瀏覽器查看原文。結構化的 XML 也讓 embedding 切片更精準——以「條」為單位而非任意字數切片。
